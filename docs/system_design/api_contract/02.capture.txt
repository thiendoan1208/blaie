API Contract - 02. Capture text (asynchronous processing)

Base path: /api/v1/captures

Text capture is accepted synchronously and classified asynchronously. PostgreSQL is the source of truth for
captures, jobs and Inbox items. Redis Stream only transports job identifiers. A capture never uses a pending
status: processing means accepted and not terminal; completed and failed are terminal.

Capture state machine:
  processing -> completed | failed

Job state machine:
  queued -> processing -> completed
  processing -> retry_wait -> queued
  processing | retry_wait -> dead

Durability and delivery semantics:
- PostgreSQL processing_jobs is authoritative. Redis Stream is only an at-least-once wake-up transport.
- A queued job records a monotonic dispatchGeneration and its next reconciliation time. If its Redis record is
  lost after outbox completion, queued-job reconciliation publishes a newer generation.
- A worker claims only the current queued dispatch generation. Duplicate and superseded Redis records therefore
  cannot create another AI attempt or duplicate Inbox items.
- Transactional outbox recovery repairs event-listener/Redis publish failures. Queued-job reconciliation repairs a
  different gap: a queued database job whose already-published Redis record was later lost.
- After the database result transaction commits, the worker acknowledges and deletes that exact Redis record.
  Producers do not apply stream MAXLEN trimming because it could remove unprocessed work.

Authentication and ownership:
- The user must be authenticated and email verified.
- user_id always comes from the current authenticated user. Clients cannot submit userId.
- Every read and retry query filters by both resource id and current user id.
- A missing or foreign capture returns the same CAPTURE_NOT_FOUND response.

----------------------------------------------------------------------

POST /api/v1/captures/text

Headers:
  Idempotency-Key: UUID (required)
  Content-Type: application/json

Request:
{
  "text": "string (required, min 1, max 10000)"
}

Behavior:
- Validate auth, email verification, capture acceptance, rate limit, body and idempotency key.
- If asynchronous capture acceptance is disabled for deployment/drain mode, reject with
  CAPTURE_PROCESSING_UNAVAILABLE before creating a capture, job or idempotency record.
- The rate-limit subject combines HMAC-SHA-256 pseudonyms of the current user id and client IP. If its Redis backend
  is unavailable, reject with SERVICE_UNAVAILABLE instead of creating an unprotected AI job.
- Resolve an existing Idempotency-Key before capacity admission. A same-payload replay still returns its existing
  capture even while the user or system is at capacity; a conflicting replay still returns IDEMPOTENCY_KEY_REUSED.
- For a new request, serialize admission in PostgreSQL and count queued, processing and retry_wait jobs. Reject with
  TOO_MANY_ACTIVE_JOBS when the current user is at their active-job limit. Reject with
  CAPTURE_PROCESSING_OVERLOADED when the global active-job limit is reached or the oldest queued work exceeds its
  configured age threshold. Both responses include Retry-After and create no capture, key, job or outbox event.
- After admission succeeds, create the processing capture, queued processing job, idempotency record and
  transactional application event in the same database transaction.
- Do not call the AI provider in the HTTP request.
- Reusing the same key with the same normalized request returns the existing capture.
- Reusing the same key with another request returns IDEMPOTENCY_KEY_REUSED.

Response 202:
{
  "data": {
    "id": "uuid",
    "originalText": "Chieu nay hop luc 5h, sau do mua sua",
    "processingStatus": "processing",
    "failureCode": null,
    "canRetry": false,
    "items": [],
    "createdAt": "2026-07-16T10:00:00Z",
    "updatedAt": "2026-07-16T10:00:00Z"
  }
}

----------------------------------------------------------------------

GET /api/v1/captures/{captureId}

Returns the current user's capture. When completed, items contains every classified Inbox item. When failed,
failureCode is a safe application code. canRetry is true only when the durable processing job is dead and its
terminal failure policy explicitly permits a manual retry; failed does not imply retryable. This read remains
available when asynchronous acceptance is disabled.

Examples:
- Provider timeout/429/5xx or invalid structured output exhausted its automatic attempts: canRetry = true.
- Provider configuration/rejection failure: canRetry = true so the user can retry after the provider is fixed.
- Sensitive credential or content-policy rejection: canRetry = false because replaying the same original text is
  unsafe or deterministically fails again.
- A failed legacy capture without a processing job: canRetry = false.

----------------------------------------------------------------------

GET /api/v1/captures?status=processing&limit=20

Returns recent processing captures owned by the current user. This endpoint restores UI state after reload,
reconnect or opening another device. status currently accepts processing only. limit is 1..50. This read remains
available when asynchronous acceptance is disabled.

----------------------------------------------------------------------

POST /api/v1/captures/{captureId}/retry

Requeues a failed capture owned by the current user only when its durable terminal policy permits manual retry.
The same capture id and original text are reused. The capture becomes processing, canRetry becomes false and the
existing logical job starts a new retry generation. The policy is checked while the capture/job rows are locked;
a rejected retry does not delete items, mutate retryGeneration or publish another job. If asynchronous acceptance
is disabled, reject with CAPTURE_PROCESSING_UNAVAILABLE before changing capture or job state.

A permitted dead job must pass the same serialized active-job and queue-health admission checks before it is
requeued. Capacity rejection leaves the capture failed, leaves the retry generation unchanged and publishes no
event. Ownership and retry-policy checks retain precedence, so a missing/foreign capture still returns 404 and a
non-retryable capture still returns 409 without disclosing queue state.

A dead job whose failure is content-terminal (for example sensitive_credential_detected) cannot be retried with
the same original text and returns CAPTURE_NOT_RETRYABLE. Missing/foreign captures continue to return the same
CAPTURE_NOT_FOUND response, so retry policy does not disclose ownership.

Response 202: Capture response with processingStatus = processing.

Errors:
  400  IDEMPOTENCY_KEY_REQUIRED | IDEMPOTENCY_KEY_INVALID
  403  EMAIL_NOT_VERIFIED
  404  CAPTURE_NOT_FOUND
  409  IDEMPOTENCY_KEY_REUSED
       CAPTURE_NOT_RETRYABLE - capture/job is not terminal or its terminal policy forbids manual retry
  422  VALIDATION_ERROR
  429  RATE_LIMITED
       TOO_MANY_ACTIVE_JOBS - the current user already has the configured number of active jobs
  503  CAPTURE_PROCESSING_UNAVAILABLE - deployment is not accepting new asynchronous capture work
       CAPTURE_PROCESSING_OVERLOADED - global queue depth or oldest queued age has crossed its admission threshold
       SERVICE_UNAVAILABLE - a required fail-closed dependency such as the capture rate-limit backend is unavailable

RATE_LIMITED, TOO_MANY_ACTIVE_JOBS and CAPTURE_PROCESSING_OVERLOADED include Retry-After. The first is short-term
HTTP abuse protection; the latter two are asynchronous-work backpressure. None is a billing or subscription quota.
