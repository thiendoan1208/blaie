API Contract - 02. Capture text (asynchronous processing)

Base path: /api/v1/captures

Text capture is accepted synchronously and classified asynchronously. PostgreSQL is the source of truth for
captures, jobs and Inbox items. Redis Stream only transports job identifiers. A capture never uses a pending
status: processing means accepted and not terminal; completed and failed are terminal.

Capture state machine:
  processing -> completed | failed

Job state machine:
  queued -> processing -> completed
  processing -> retry_wait -> queued
  processing | retry_wait -> dead

Authentication and ownership:
- The user must be authenticated and email verified.
- user_id always comes from the current authenticated user. Clients cannot submit userId.
- Every read and retry query filters by both resource id and current user id.
- A missing or foreign capture returns the same CAPTURE_NOT_FOUND response.

----------------------------------------------------------------------

POST /api/v1/captures/text

Headers:
  Idempotency-Key: UUID (required)
  Content-Type: application/json

Request:
{
  "text": "string (required, min 1, max 10000)"
}

Behavior:
- Validate auth, email verification, rate limit, body and idempotency key.
- The rate-limit subject combines hashed current user id and client IP. If its Redis backend is unavailable, reject
  with SERVICE_UNAVAILABLE instead of creating an unprotected AI job.
- In one database transaction, create the processing capture, queued processing job, idempotency record and
  transactional application event.
- Do not call the AI provider in the HTTP request.
- Reusing the same key with the same normalized request returns the existing capture.
- Reusing the same key with another request returns IDEMPOTENCY_KEY_REUSED.

Response 202:
{
  "data": {
    "id": "uuid",
    "originalText": "Chieu nay hop luc 5h, sau do mua sua",
    "processingStatus": "processing",
    "failureCode": null,
    "canRetry": false,
    "items": [],
    "createdAt": "2026-07-16T10:00:00Z",
    "updatedAt": "2026-07-16T10:00:00Z"
  }
}

----------------------------------------------------------------------

GET /api/v1/captures/{captureId}

Returns the current user's capture. When completed, items contains every classified Inbox item. When failed,
failureCode is a safe application code and canRetry is true.

----------------------------------------------------------------------

GET /api/v1/captures?status=processing&limit=20

Returns recent processing captures owned by the current user. This endpoint restores UI state after reload,
reconnect or opening another device. status currently accepts processing only. limit is 1..50.

----------------------------------------------------------------------

POST /api/v1/captures/{captureId}/retry

Requeues a failed capture owned by the current user. The same capture id and original text are reused. The
capture becomes processing and the existing logical job starts a new retry generation.

Response 202: Capture response with processingStatus = processing.

Errors:
  400  IDEMPOTENCY_KEY_REQUIRED | IDEMPOTENCY_KEY_INVALID
  403  EMAIL_NOT_VERIFIED
  404  CAPTURE_NOT_FOUND
  409  IDEMPOTENCY_KEY_REUSED | CAPTURE_NOT_RETRYABLE
  422  VALIDATION_ERROR
  429  RATE_LIMITED
  503  SERVICE_UNAVAILABLE
