# ADR-003: Use Layered Modular Architecture with Light Hexagonal Boundaries

## Status

Accepted

## Context

Backend Blaie là một Spring Boot modular monolith. Hệ thống có nhiều domain/module:

- auth
- user
- capture
- item
- file
- ai-processing
- notification
- admin

Nếu dùng layered architecture truyền thống ở cấp toàn project:

```text
src
├── controller
├── service
├── repository
├── entity
└── dto
```

thì ban đầu dễ hiểu, nhưng khi nhiều module xuất hiện, code sẽ bị chia theo technical
layer thay vì business domain. Muốn sửa feature `capture`, dev phải nhảy qua nhiều folder:
controller, service, repository, entity, dto. Điều này làm module boundary mờ dần và dễ
dẫn tới service/repository gọi chéo lung tung.

Ngược lại, nếu áp dụng full Hexagonal/Clean Architecture cho mọi endpoint CRUD nhỏ, MVP sẽ
bị nặng vì phải tạo quá nhiều interface, mapper, port, adapter và use case không cần thiết.

## Decision

Use a layered modular architecture with light hexagonal boundaries.

Nói chính xác hơn:

- System/backend level: modular monolith.
- Module level: layered architecture by default.
- External dependency/business-critical flow: use light hexagonal ports and adapters.

Mục tiêu là giữ backend vừa dễ build cho MVP, vừa đủ sạch để phát triển lâu dài.

## Proposed package structure

Tổng thể project:

```text
com.blaie
├── BlaieApplication.java
│
├── core
│   ├── config
│   ├── error
│   ├── response
│   ├── request
│   ├── logging
│   └── security
│
├── auth
├── user
├── capture
├── item
├── file
├── ai
├── notification
├── admin
└── shared
```

Một business module nên có cấu trúc:

```text
capture
├── api
│   ├── CaptureController.java
│   ├── request
│   └── response
│
├── application
│   ├── CaptureService.java
│   ├── command
│   ├── result
│   └── port
│
├── domain
│   ├── CaptureItem.java
│   ├── CaptureSourceType.java
│   └── ProcessingStatus.java
│
└── infrastructure
    ├── persistence
    └── adapter
```

Ý nghĩa:

- `api`: REST controller, request DTO, response DTO.
- `application`: use case/service, command/result, port interface.
- `domain`: business model/rule nếu module có logic đáng kể.
- `infrastructure`: JPA adapter, queue adapter, storage adapter, external API adapter.

Không phải module nào cũng cần đầy đủ mọi folder. CRUD đơn giản có thể nhẹ hơn, nhưng
module có business flow quan trọng nên theo cấu trúc này.

## Request flow

Flow chung cho một HTTP request:

```text
request
-> request context / requestId
-> authentication
-> coarse authorization
-> validation
-> controller
-> service/use case
-> load resource
-> object-level authorization
-> domain/business logic
-> repository / external adapter
-> response DTO
-> centralized error handler if error
```

Ví dụ user capture text:

```text
Web/Mobile
  |
  v
POST /api/v1/captures/text
  |
  v
CaptureController
  |
  v
CaptureService
  |
  +--> CaptureItemRepository port
  |       |
  |       v
  |   JpaCaptureItemRepositoryAdapter
  |       |
  |       v
  |   PostgreSQL
  |
  +--> ProcessingJobPublisher port
          |
          v
      RedisProcessingJobPublisher
          |
          v
      Job Queue
```

Capture flow:

1. Controller nhận request và map sang command.
2. Service lấy current user từ security context.
3. Service tạo CaptureItem với processing status = pending.
4. Service lưu item qua repository port.
5. Service publish processing job qua job publisher port.
6. API trả response ngay cho client, không chờ AI.

## Background worker flow

AI processing worker cũng dùng cùng tư duy port/adapter:

```text
Job Queue
  |
  v
AiProcessingWorker
  |
  v
AiProcessingService
  |
  +--> CaptureItemRepository port
  +--> AiProviderPort
  +--> FileStoragePort
          |
          v
      OpenAiAdapter / GeminiAdapter / StorageAdapter
```

Worker flow:

1. Worker consume job từ queue.
2. Service load item.
3. Service chọn pipeline theo source type: text, image, voice/audio.
4. Service gọi AI provider qua `AiProviderPort`.
5. Service cập nhật title, summary, category, metadata, confidence.
6. Service không overwrite manual edits trừ khi user explicitly reprocess.
7. Service mark processing status completed hoặc failed.

## Error handling boundary

Backend phải có centralized error handling bằng Spring Boot, ví dụ:

- custom application exception.
- error code chuẩn.
- `@RestControllerAdvice`.
- response format thống nhất.

Controller không parse raw database error hoặc external API error. Repository/adapter có thể
map raw technical error thành application-level error. Service quyết định business meaning.

Response lỗi nên theo dạng:

```json
{
  "code": "CAPTURE_ITEM_NOT_FOUND",
  "message": "Capture item not found",
  "requestId": "req_123"
}
```

Validation error:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Validation failed",
  "errors": {
    "title": ["Title is required"]
  },
  "requestId": "req_123"
}
```

Rule:

- Controller không catch lỗi tùy tiện.
- Service throw lỗi có nghĩa nghiệp vụ.
- Repository/adapter không leak raw JPA/SQL/SDK error ra client.
- Global exception handler serialize error body một lần.
- Internal log có chi tiết hơn user response, nhưng không log secret/token/password.

## Authorization boundary

Frontend chỉ render UX theo quyền. Backend là security boundary thật.

Backend authorization nên đi theo nhiều lớp:

```text
authenticate
-> resolve principal/session
-> coarse authorization
-> load resource
-> object-level authorization
-> property-level filtering if needed
-> execute business logic
```

Trong V1, quyền tối thiểu cần đảm bảo:

- User chỉ được truy cập item/file/tag/metadata của chính mình.
- Admin API yêu cầu admin permission.
- Resource ownership phải check ở backend.
- Không nhận `userId`, `role`, `ownerId` từ client để quyết định quyền.

Nên có authorization module/service riêng thay vì rải `if role == admin` khắp controller.

Ví dụ API tư duy:

```text
authorize(principal, action, resource, context)
```

Trong đó:

- principal: current user/session/roles.
- action: capture.create, item.read, item.update, admin.job.retry.
- resource: object thật đã load từ DB nếu cần.
- context: tenant/workspace/plan/feature flag nếu sau này có.

## Response and DTO boundary

API không trả thẳng JPA entity.

Rule:

- Request DTO chỉ chứa field client được phép gửi.
- Response DTO chỉ chứa field client cần thấy.
- Sensitive/internal fields không tự động serialize ra ngoài.
- Property-level permission nếu có phải xử lý ở serializer/DTO/service layer.

Điều này giúp tránh:

- mass assignment.
- leak internal fields.
- frontend phụ thuộc trực tiếp vào DB schema.

## SOLID principles

Architecture này hỗ trợ SOLID theo cách thực dụng:

### Single Responsibility Principle

- Controller xử lý HTTP mapping.
- Service/use case xử lý flow nghiệp vụ.
- Repository xử lý data access.
- Adapter xử lý external system.
- Error handler xử lý serialization lỗi.

### Open/Closed Principle

Có thể thêm provider mới bằng adapter mới:

```text
AiProviderPort
  ^-- OpenAiAdapter
  ^-- GeminiAdapter
```

Core service không cần sửa nhiều khi đổi provider.

### Liskov Substitution Principle

Các adapter implement cùng port phải thay thế được nhau nếu giữ đúng contract.

### Interface Segregation Principle

Port nên nhỏ theo use case, không tạo interface khổng lồ như `ExternalServicePort` làm mọi
thứ.

### Dependency Inversion Principle

Application service phụ thuộc port/interface, không phụ thuộc trực tiếp vào SDK, Redis,
S3/R2, queue implementation hoặc AI provider cụ thể.

## Why not traditional layered architecture?

Layer truyền thống toàn project có ưu điểm là đơn giản, nhưng có các vấn đề:

- Code chia theo kỹ thuật, không theo business.
- Folder controller/service/repository phình to khi nhiều domain.
- Module boundary mờ.
- Service dễ gọi chéo repository của module khác.
- Khó tách module sau này.
- External SDK dễ lọt thẳng vào business service.

Layered modular architecture giải quyết bằng cách gom code theo domain:

```text
capture/
item/
ai/
file/
admin/
```

Muốn sửa capture thì vào module capture. Muốn sửa AI processing thì vào module ai. Code
liên quan nằm gần nhau hơn.

## Why not full hexagonal architecture?

Full hexagonal/clean architecture cho toàn bộ codebase có thể rất sạch, nhưng V1 là MVP.
Nếu CRUD đơn giản cũng phải tạo đủ port, adapter, mapper, use case, domain model thì tốc
độ phát triển sẽ chậm và code có nhiều ceremony.

Vì vậy chọn hướng practical:

- Layered by default.
- Hexagonal where it matters.

Chỗ nên dùng port/adapter:

- AI provider.
- Object storage.
- Queue/job publisher.
- Notification provider.
- Payment provider sau này.
- Business flow quan trọng cần test kỹ.

Chỗ có thể dùng layered đơn giản:

- CRUD ít logic.
- Admin list đơn giản.
- Basic lookup.
- Simple tag management.

## Enforced Backend Boundary Rules

Sau khi backend bắt đầu có auth, rate-limit, email verification, password reset và OAuth,
ADR này được siết lại bằng ArchUnit để convention không chỉ phụ thuộc vào code review.

Rule chính:

- `domain` phải độc lập với adapter/API/framework:
  - không phụ thuộc `api`.
  - không phụ thuộc `infrastructure`.
  - không phụ thuộc JPA/Spring Data/Spring Web.
- `application` không biết implementation adapter:
  - không import `infrastructure`.
  - không import JPA entity/repository.
  - không import Spring Data/Spring Security/Spring Web.
  - application service chỉ dùng command/result/port/domain/core primitive.
- `application.port` là contract sạch:
  - port không phụ thuộc `infrastructure`.
  - port dùng interface/DTO view nhẹ, ví dụ `AuthUserView`, `AuthTokenPort`,
    `EmailSettingsPort`.
- `api` không phụ thuộc implementation infrastructure nội bộ:
  - controller không inject trực tiếp service infra như cookie service/OAuth state service.
  - controller gọi application service hoặc application port.
- `infrastructure` không phụ thuộc `api`:
  - adapter có thể implement application port.
  - adapter có thể dùng JPA, Redis, SDK, Spring Security, email provider.
  - adapter không import controller/request/response DTO.
- `core` không phụ thuộc business module:
  - `core` không import `auth`, `authz`, `capture`, `inbox`, ...
  - `core` chỉ chứa primitive dùng chung.
- `core.ratelimit` không phụ thuộc auth implementation:
  - rate-limit generic không import `AuthCookieService` hoặc auth infrastructure.
  - nếu cần tên cookie hoặc primitive chung, đưa vào core-level constant/config.

Dependency direction mong muốn:

```text
api -> application -> domain / application.port
infrastructure -> application.port + domain
feature module -> core primitive
core -> no business module
```

Các rule này được enforce bằng `ArchitectureRulesTest`. Khi có import sai hướng, test
phải fail sớm.

## Security Boundary After Auth Refactor

Security được chia làm hai phần rõ hơn.

`core.security` chỉ giữ primitive dùng chung:

- `SecurityCorsProperties`: cấu hình CORS chung.
- `SecurityErrorResponseWriter`: writer JSON lỗi security dùng chung.
- `AppAuthenticationEntryPoint`: xử lý lỗi unauthenticated, thường là `401`.
- `AppAccessDeniedHandler`: xử lý lỗi forbidden, thường là `403`.
- `CurrentUser`: model current user tối giản.
- `CurrentUserHolder`: holder current user trong request.
- `AuthCookieNames`: tên cookie auth dùng chung khi code core/rate-limit cần đọc cookie
  name mà không import auth implementation.

`auth.infrastructure.security` giữ implementation thật của auth/security runtime:

- `SecurityConfig`: Spring Security filter chain, CSRF, CORS, permit/authenticated paths,
  password encoder.
- `AuthProperties`: config auth như token TTL, cookie flags, secrets.
- `AuthProductionSecurityValidator`: validate config bảo mật khi chạy production.
- `AuthTokenService`: issue/parse/hash token, implement `AuthTokenPort`.
- `AuthCookieService`: build access/refresh cookie, implement `WebAuthCookiePort`.
- `AuthRequestFilter`: đọc Bearer/cookie access token, parse token, load user, set
  `CurrentUser`.
- `EmailVerificationRequiredFilter`: chặn request cần verified email nếu user chưa verify.
- `BearerTokenResolver`: parse `Authorization: Bearer ...`.
- `AccessTokenClaims`: claims sau khi parse access token.
- `PasswordHasherAdapter`: adapter bọc Spring `PasswordEncoder`, implement
  `PasswordHasherPort`.

Lý do `SecurityConfig` nằm trong `auth.infrastructure.security`:

- `SecurityConfig` không phải shared core primitive.
- Nó wire các filter/properties cụ thể của auth.
- Nó biết các path auth public như login/register/refresh/google/password reset.
- Nếu để trong `core.security`, dependency sẽ bị ngược thành `core -> auth.infrastructure`.
- Đặt trong infrastructure giúp composition root biết các mảnh cần ghép, còn `core` vẫn
  không biết module nghiệp vụ nào đang dùng nó.

Việc move này không đổi runtime behavior:

- URL API không đổi.
- Filter order không đổi.
- CSRF/CORS config không đổi.
- Cookie name/flag behavior không đổi.
- Login/register/refresh/logout/OAuth/password reset/email verification không đổi.
- Spring vẫn scan được vì package mới vẫn nằm dưới root app package.

## How Future Features Use Auth And Security

Feature mới như `inbox`, `capture`, `ai-processing`, `billing`, `admin` dùng kết quả của
auth, không dùng implementation của auth.

Feature application/service được phép dùng:

```java
import com.blaie.blaie_be.core.security.CurrentUser;
import com.blaie.blaie_be.core.security.CurrentUserHolder;
import com.blaie.blaie_be.authz.application.AuthorizationService;
import com.blaie.blaie_be.authz.domain.PermissionAction;
```

Feature không nên import:

```java
com.blaie.blaie_be.auth.infrastructure.security.AuthTokenService
com.blaie.blaie_be.auth.infrastructure.security.AuthCookieService
com.blaie.blaie_be.auth.infrastructure.security.AuthRequestFilter
com.blaie.blaie_be.auth.infrastructure.security.SecurityConfig
com.blaie.blaie_be.auth.infrastructure.persistence.UserEntity
```

Flow mong muốn:

```text
Request
  -> SecurityConfig/filter chain
  -> AuthRequestFilter authenticates user
  -> CurrentUserHolder is set
  -> Feature controller
  -> Feature application service
  -> CurrentUserHolder + AuthorizationService
  -> Feature repository/adapter
```

Auth/security chỉ trả lời các câu hỏi cấp request:

- Request này là ai?
- Token/cookie hợp lệ không?
- User còn active không?
- User đã verify email chưa nếu endpoint yêu cầu?
- Current user context là gì?

Feature module tự xử lý business authorization của mình:

- ownership.
- object-level permission.
- plan/quota/cost rule.
- capture/inbox/AI-specific business rule.

Ví dụ:

- `InboxService` check ownership và quyền đọc/sửa item qua `AuthorizationService`.
- `CaptureService` check quyền tạo capture, quota upload/process và ownership.
- `AI processing` gọi AI provider qua port/adapter riêng, không gọi trực tiếp SDK từ
  application service.

`SecurityConfig` chỉ xử lý route/filter-level concern:

- endpoint public mới thì thêm permit rule.
- endpoint authenticated bình thường đi theo `.anyRequest().authenticated()`.
- endpoint cần skip email verification thì thêm vào email-verification path policy.

Không đưa rule nghiệp vụ của feature vào `AuthRequestFilter` hoặc `SecurityConfig`.

## Rate Limit Boundary

Rate-limit generic nằm trong `core.ratelimit`, nhưng không được biết implementation auth.

Rule:

- HTTP rate-limit filter xử lý request-level abuse chung.
- Business guard nằm trong application service nếu rule gắn với nghiệp vụ cụ thể.
- Không lưu identifier raw như email nếu không cần; dùng normalized hash.
- Endpoint cost cao như upload/AI processing cần policy riêng theo user/IP/plan/quota.

Ví dụ tương lai:

- `capture-upload`: limit theo user id + IP.
- `ai-process`: limit theo user id + plan/quota/cost.
- `inbox-read`: limit rộng hơn, chủ yếu chống spam.
- `admin`: limit theo admin user + IP.

Nếu policy resolver phình to, tách thành registry hoặc resolver theo module để tránh một
class biết toàn bộ feature.

## Verification Expectations For Large Refactors

Refactor kiến trúc lớn không thể chắc chắn tuyệt đối chỉ bằng tự động test. Tuy nhiên,
backend phải giảm rủi ro theo nhiều lớp:

- compile/test-compile sạch.
- ArchUnit pass để xác nhận boundary.
- unit test application pass.
- Spring context test pass để xác nhận bean wiring.
- integration/web flow test pass qua HTTP layer, security filter, cookies, CSRF, DB,
  Redis/Testcontainers và migration.
- Flyway migration chạy sạch trên database test.
- full suite pass trước khi xem refactor là an toàn để manual QA.

Manual QA vẫn cần cho các flow có browser/FE/provider thật:

- register.
- login đúng/sai và rate-limit.
- refresh session sau reload.
- logout.
- password reset.
- Google OAuth thật trên browser.
- CSRF từ frontend.
- email verification resend cooldown/quota.
- cookie flags trong browser devtools.

Khi refactor boundary, cố giữ nguyên behavior surface:

- không đổi endpoint URL.
- không đổi filter order nếu không có lý do rõ.
- không đổi response body/error contract.
- không đổi cookie name.
- không đổi migration đã publish.
- ưu tiên đổi dependency direction hơn là đổi behavior nghiệp vụ.

## Consequences

Ưu điểm:

- Code đọc theo business domain rõ hơn.
- Dễ hiểu, dễ onboard hơn full clean architecture.
- Dễ test use case quan trọng.
- External provider/storage/queue dễ đổi hơn.
- Hợp với Spring Boot và MVP.
- Có đường tiến hóa lên microservices nếu module boundary rõ.

Trade-off:

- Nhiều structure hơn layered architecture truyền thống.
- Cần discipline để không phá module boundary.
- Có thể inconsistent nếu team không thống nhất convention.
- Vẫn cần quyết định khi nào dùng port/adapter, khi nào dùng layered đơn giản.

Mitigation:

- Viết convention rõ.
- Review dependency giữa module.
- Chỉ tạo port/adapter khi có external dependency hoặc business flow quan trọng.
- Không để controller chứa business logic nặng.
- Không để service gọi trực tiếp SDK/provider.
- Không trả JPA entity trực tiếp ra API.

## Alternatives Considered

### Traditional layered architecture

Dễ bắt đầu nhưng dễ rối khi số domain tăng.

### Full hexagonal architecture

Sạch hơn nhưng dễ over-engineer MVP.

### Microservices with hexagonal services

Không chọn ở V1 vì đã chốt modular monolith trong ADR-001. Microservices sẽ tăng chi phí
vận hành, tracing, service contract và consistency trước khi product/domain boundary đủ rõ.
