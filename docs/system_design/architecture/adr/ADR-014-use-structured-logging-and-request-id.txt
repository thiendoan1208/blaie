# ADR-014: Use Structured Logging and Request ID

## Status

Accepted

## Context

Blaie có sync request flow và async worker flow. Khi production lỗi, cần tìm được request
hoặc job liên quan. AI processing có thể fail vì provider, timeout, bad input hoặc worker
error. Without structured logs and request/job id, debugging sẽ rất khó.

## Decision

Use structured logging and requestId/jobId correlation.

Backend API:

- Create or forward X-Request-ID after validating a bounded safe character set.
- Include requestId in response header.
- Log method, path, userId if available, status, latencyMs, errorCode.

Worker:

- Persist the original HTTP request ID on `processing_jobs` so correlation survives outbox
  resubmission, Redis loss/reconciliation, process restart and delayed execution.
- Carry requestId, eventId, jobId, captureId and dispatchGeneration through the transactional
  event and Redis wake-up record. Older outbox/Redis records without requestId fall back to a
  safe event/job ID and remain processable.
- Restore the persisted requestId after the database claim, making PostgreSQL authoritative
  when transport metadata is missing or stale.
- Add a unique providerAttemptId for each real provider route attempt.
- Log jobId, captureId, attempts/generations, processing state and safe errorCode. Do not put
  original text, provider payloads or credentials in correlation context.

Context propagation:

- HTTP, outbox publisher, Redis worker, claimed job and provider attempt each use a scoped MDC
  context that restores the caller's previous values on exit. Background execution must not
  call a global `MDC.clear()` that can erase an outer context.
- This decision provides durable log correlation. It does not claim end-to-end distributed
  tracing; OpenTelemetry trace/span propagation can be added later without changing the
  persisted request ID contract.

Metrics:

- Correlation identifiers never become metric labels. Micrometer labels are restricted to
  bounded states, outcomes, failure classes, sources and configured provider IDs.
- Admin mutations log actor ID, request ID, job ID, action and a safe state transition. Durable
  admin audit storage and its retention policy are handled by the privacy/retention work.

Error response:

- Return safe error body with code, message and traceId/requestId.

## Consequences

Ưu điểm:

- Dễ debug production issue.
- User/frontend có thể báo traceId.
- Admin/job monitoring dễ hơn.
- Chuẩn bị tốt cho metrics/tracing sau này.

Trade-off:

- Cần logging convention.
- Log storage có cost.
- Có nguy cơ log nhầm sensitive data.

Mitigation:

- Không log password, token, secret, raw Authorization header.
- Không log full private content mặc định.
- Dùng log level theo environment.
- Có retention policy khi production.

## Alternatives Considered

### Plain console logs

Không đủ cho debugging production-like system.
