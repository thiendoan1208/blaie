ADR-019 - Use durable asynchronous processing for text capture

Status: Accepted

## Context

The synchronous text capture request waits for the external AI provider. Provider latency and availability then
consume web request threads and directly determine API latency. Client retries after a lost response can also
create duplicate captures.

## Decision

- POST /api/v1/captures/text returns 202 after committing a processing capture and queued job.
- PostgreSQL processing_jobs is the durable execution source of truth.
- Spring Modulith event_publication is the transactional outbox.
- Redis Stream transports job identifiers using at-least-once delivery.
- Redis is a wake-up transport, not the authoritative job state. Losing a Redis record must not lose the job.
- Workers are idempotent and acknowledge Redis only after the result transaction commits.
- Capture states are processing/completed/failed. Job states are queued/processing/retry_wait/completed/dead.
- Automatic retries use backoff; leased processing jobs are recovered after the lease expires.
- HTTP Idempotency-Key prevents duplicate captures when a response is lost.
- Raw capture text stays in PostgreSQL and is never included in the Redis message.
- A provider router selects the configured provider and can try ordered fallback providers when the failure class
  permits provider fallback. Provider fallback and automatic job retry are independent policy decisions.
- Credential-like secrets are rejected before any provider call and are not automatically retried.
- Provider timeout/429/5xx and invalid structured output permit fallback and automatic retry. Provider-local
  configuration/rejection failures permit fallback but are terminal for the current job attempt. Content-policy
  failures stop fallback and are terminal.
- Capture submit/retry rate limits use HMAC-SHA-256 pseudonyms of user id + IP and fail closed when Redis is
  unavailable. The security-chain registration is the only servlet registration for these filters, preserving
  authentication -> rate limit -> email-verification ordering.
- PostgreSQL serializes admission for new and manually retried jobs. Per-user active jobs, global active jobs and
  oldest queued age bound how much work the system accepts before it writes another capture.
- Redis provides a distributed leased semaphore per AI provider. Semaphore saturation waits for that provider and
  does not itself trigger fallback or consume another provider attempt.
- Operational logs include job/capture ids, attempts, safe error codes, provider/model and latency, never raw text.
- Quota/payment usage accounting is explicitly deferred. Capture-specific Redis rate limiting is included.

## Failure and retry semantics

Text classification failures use a closed failure taxonomy instead of one overloaded retryable boolean. Each class
decides three different questions: whether the router may try the next provider, whether the job may retry
automatically, and whether a user may manually replay a dead job later.

| Failure class | Typical examples | Provider fallback | Automatic retry | Manual retry after dead |
| --- | --- | --- | --- | --- |
| CONTENT_TERMINAL | credential detected, content policy blocked | no | no | no |
| PROVIDER_TERMINAL | provider key/configuration missing, provider-local 4xx rejection | yes | no | yes, after the provider/configuration is fixed |
| PROVIDER_RETRYABLE | timeout, 408/429/5xx, invalid or truncated structured output | yes | yes with backoff | yes after attempts are exhausted |
| SYSTEM_RETRYABLE | unexpected processing/infrastructure error, expired final lease | no | yes with backoff | yes after attempts are exhausted |

The safe failure code remains the public diagnostic value, while the failure class is the execution policy. The
latest terminal failure class is persisted on processing_jobs instead of inferring retryability merely from
capture.processing_status = failed. API canRetry is true only for an owned failed capture whose job is dead and
whose effective failure class permits manual retry. Starting a permitted retry clears the active failure metadata
while the new retry generation runs. Replaying content-terminal input is rejected with CAPTURE_NOT_RETRYABLE and
causes no state change or publication.

During the rolling upgrade, known safe failure codes take precedence and map deterministically to a failure class;
existing jobs are backfilled with the same mapping. A V12 binary can still omit the nullable class column. For an
unknown code without stored class, the compatibility fallback is SYSTEM_RETRYABLE so an infrastructure failure
does not become permanently stuck. Once V12 binaries are drained, a later migration may make the failure metadata
strictly non-null for terminal jobs. Failed legacy captures without a durable job remain non-retryable.

## Durable dispatch and recovery

The durable flow has two different delivery gaps and therefore two different repair loops:

1. Transactional outbox recovery covers failure before Redis dispatch completes. Creating or redispatching a job
   and recording TextCaptureQueuedEvent happen in the same PostgreSQL transaction. If the event listener or Redis
   XADD fails, Spring Modulith leaves event_publication incomplete and later resubmits that publication.
2. Queued-job reconciliation covers failure after the outbox publication is already complete. A Redis record can
   still disappear or be deleted operationally. A queued processing_jobs row keeps last_dispatched_at,
   next_dispatch_at and a monotonically increasing dispatch_generation. When next_dispatch_at is due, recovery
   republishes the job and advances its dispatch generation.

Outbox recovery is intentionally attempted before queued reconciliation becomes due. The first dispatch retry
delay must therefore be greater than outbox_recovery_age. This avoids both repair loops immediately producing the
same wake-up record while still allowing eventual recovery when the Redis record itself is lost.

Each Redis record carries jobId, captureId, eventId and dispatchGeneration, never the raw user text. A worker may
claim the PostgreSQL job only when it is still queued and the message dispatchGeneration equals the current job
generation. This database compare-and-set is the idempotency boundary: duplicate or superseded Redis messages
cannot run the AI attempt again. Attempt ownership is also fenced by worker id, attempt count and retry generation,
so a worker whose lease expired cannot commit over a newer attempt.

Delivery is at least once. After a terminal processing transaction or a harmless duplicate has been resolved, the
worker performs XACK followed by XDEL for that exact record inside one Redis Lua script. The server-side operation
closes the client-disconnect gap between two separate commands and works on Redis 7 without Redis 8 XACKDEL. An
ambiguous connection failure is safe: PostgreSQL already contains the processing result, redelivery is idempotent,
and rerunning the Lua script is harmless.

The producer never uses MAXLEN trimming because producer-side length trimming can delete a record that no consumer
has processed. The recovery role periodically applies approximate XTRIM MINID using an age cutoff; defaults are a
24-hour retention and a one-hour cleanup interval, both configurable. PostgreSQL queued-job reconciliation remains
the recovery path if an old Redis wake-up record is removed. Exact-record XDEL assumes the capture stream is
dedicated to one consumer group. A future second consumer group requires group-aware retention instead of deleting
immediately after the first group acknowledges.

## Startup roles and admission control

The legacy blaie.capture.processing.enabled switch is deprecated. It remains a compatibility fallback for one
release: each role-specific property defaults to the legacy value, while an explicitly configured role-specific
property takes precedence.

Role-specific startup properties:

- accept-async-enabled controls whether POST text capture and manual retry accept new asynchronous work.
- publisher-enabled installs the transactional outbox listener that writes wake-up records to Redis.
- worker-enabled installs the Redis consumer and AI job executor.
- recovery-enabled installs stale-lease recovery, ready-retry dispatch, queued-job reconciliation and outbox
  resubmission.

Valid deployment combinations include:

- all-in-one: accept=true, publisher=true, worker=true, recovery=true.
- API node: accept=true, publisher=true, worker=false; recovery may run on a designated recovery node instead.
- worker node: accept=false, publisher=false, worker=true, recovery=false.
- recovery node: accept=false, publisher=true, worker=false, recovery=true.
- drain mode: accept=false, publisher=true, worker=true, recovery=true, so existing work finishes while new work is
  rejected.
- fully disabled: all four flags are false.

Startup validation rejects accept=true with publisher=false and recovery=true with publisher=false. Cross-instance
availability cannot be validated by one process, so deployment configuration must still guarantee at least one
worker and one recovery loop cluster-wide whenever asynchronous capture is accepted.

When acceptance is disabled, POST /api/v1/captures/text and POST /api/v1/captures/{captureId}/retry fail with 503
CAPTURE_PROCESSING_UNAVAILABLE before capture/job persistence. Read endpoints remain available, including capture
status restoration and Inbox reads. The flags are startup configuration, not runtime traffic switches.

## Backpressure and provider concurrency

Admission and HTTP rate limiting solve different problems. Redis HTTP rate limits reject short request bursts. The
database admission guard bounds durable work that has already been accepted but is not terminal. Quota/payment
accounting remains deferred.

For every new logical job or permitted manual retry, the workflow locks the singleton capture-admission row inside
its transaction. While holding that lock it evaluates:

- the current user's jobs in queued, processing or retry_wait;
- all jobs in those active states; and
- the age of the oldest queued job, measured from available_at for its current queue cycle.

At the configured per-user boundary the API returns 429 TOO_MANY_ACTIVE_JOBS. At the global boundary or queued-age
threshold it returns 503 CAPTURE_PROCESSING_OVERLOADED. Both include a configurable Retry-After. The checks and the
new active transition share one transaction, preventing concurrent accepted requests from writing past the bound.
Concurrent worker completion may cause a conservative temporary rejection, but cannot create an over-admission
race. A rejection rolls back without a capture, idempotency key, job mutation or outbox publication.

Idempotency and ownership semantics take precedence over capacity. POST resolves a matching existing key again
after acquiring the admission lock, so a lost-response replay returns its original capture even when capacity is
full. A same-key payload conflict remains 409. Retry validates ownership and terminal policy before revealing queue
health, then rechecks locked state before changing the dead job to queued.

The singleton lock is added by an additive migration. During a rolling deployment, old binaries do not acquire it;
operators must drain old writers before relying on strict limits. Terminal worker transitions do not require the
lock because they only reduce active work.

Scheduled poll/recovery/cleanup triggers, database job heartbeats and provider-permit renewals use three dedicated,
separately sized scheduler pools. A slow trigger therefore cannot delay either lease-safety loop.

Before each real provider call, the router acquires a provider-specific permit from a Redis sorted set. One atomic
Lua operation uses Redis server time to remove expired owners and install a unique leased owner only when the
configured global limit has room. While the provider call is active, a dedicated bounded scheduler renews that
exact owner token with another Redis-time Lua operation. The renewal interval is at most one third of the lease,
leaving room for a delayed tick. The permit cancels renewal and releases its owner in finally; process failure stops
renewal and is recovered by lease expiry. A stale owner cannot renew or release a replacement owner's permit.

The worker's database heartbeat continues while it waits. Waiting for provider capacity is not a provider failure,
does not spill traffic into a fallback provider and does not consume an additional attempt. Real provider failures
continue to use the failure taxonomy and ordered fallback policy above. An acquire/scheduling backend failure is
treated as SYSTEM_RETRYABLE so an unbounded provider call is never started. A transient renewal failure is retried
on the next scheduled tick; if Redis reports that ownership is already lost, further renewal stops and the incident
is logged. The base lease should still exceed the normal provider timeout as a second safety margin against a
prolonged Redis or scheduler outage. Permit usage/wait/renewal metrics and dynamic provider health are deferred to
the observability and multi-provider commits.

Rate-limit Redis subjects no longer use a plain digest. A secret-keyed HMAC prevents offline enumeration of
low-entropy identifiers if Redis keys are exposed. Production should provide a dedicated
BLAIE_RATE_LIMIT_SUBJECT_HMAC_SECRET; compatibility currently falls back to the access-token secret. Rotating this
secret changes the keyspace and effectively resets counters, whose old keys expire within their configured window.
The current capture policy intentionally retains one composite user+IP bucket, fixed windows and forwarded-header
trust disabled. Independent IP/user buckets, trusted-proxy configuration and alternative algorithms require real
deployment topology and traffic evidence and are not part of this decision.

## Consequences

Capture latency no longer depends on the AI provider and jobs survive client disconnects, Redis outages and
worker restarts. PostgreSQL-backed reconciliation also lets Redis stay an operational accelerator rather than a
durability dependency. The trade-off is eventual consistency, duplicate wake-up messages by design, additional
worker/recovery operations and a frontend that must restore and poll processing captures.
